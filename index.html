NullPointerException — the #1 Java runtime error
NPEs occur when a method is called or field accessed on a null reference. Prevention: use Optional<T> instead of returning null, add null checks before dereferencing, use Objects.requireNonNull() for parameter validation, and annotate with @NonNull / @Nullable for IDE warnings. Java 14+ helpful NPE messages identify exactly which variable was null.

Resource leaks — Streams, Connections, Files
Any class implementing Closeable or AutoCloseable must be closed: InputStream, OutputStream, Connection, PreparedStatement, ResultSet, FileWriter. Always use try-with-resources (try (Resource r = new Resource())) — it guarantees close() is called even if an exception is thrown. Never rely on the garbage collector to close I/O resources.

String comparison with ==
== compares object identity (memory address), not content. "hello" == "hello" may be true due to string interning, but new String("hello") == "hello" is false. Always use .equals() for string content comparison. Use "literal".equals(variable) pattern (Yoda condition) to avoid NPE if the variable is null.

Static analysis examines Java source code or bytecode without executing it, looking for patterns that indicate bugs, security vulnerabilities, or style violations. It differs from runtime analysis (profilers, heap dump analysis) which requires running the program. Tools like SpotBugs analyze bytecode; PMD and Checkstyle analyze source code. This analyzer uses pattern matching on source code text — a lightweight form of static analysis. Full static analysis requires a complete parse tree (AST).
